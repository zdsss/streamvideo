## 一、竞品调研

### 1.1 olived.app

定位：现代化全平台直播录制工具，主打"set it and forget it"自动化体验。

**核心能力：**
- 自动监控主播上线状态，上线即开始录制，无需人工干预
- 支持 TikTok、Twitch、YouTube、Chaturbate 等 10+ 平台
- 自动归档，支持历史回放
- 任务级高级配置（每个主播独立设置录制参数）
- 完善的文档站点，支持自定义配置项

**产品理念：** 强调"录制即归档"，用户不需要管理文件，系统自动整理。

### 1.2 Downie

定位：macOS 最强视频下载工具，覆盖面极广。

**核心能力：**
- 支持 1200+ 网站，快速适配平台变化
- 4K/8K 下载，格式转换，音频提取
- 浏览器扩展一键触发下载
- 字幕下载，批量下载
- HLS HEVC + Dolby Atmos 支持
- 隐私过滤、代理支持

**产品理念：** 极简操作，拖拽 URL 即下载，不做复杂管理。

## 二、当前项目现状审计

### 2.1 已完善的功能

**录制核心**
- streamlink → yt-dlp 两级降级，抖音额外支持 Playwright+ffmpeg
- 断流检测（stall timeout 30s）+ 自动重连
- 自动分割（按大小/时长）
- 磁盘空间监控（每 60s 检查，critical 时停录）

**智能合并系统**
- 信心度评分（`_calc_merge_confidence`，`recorder.py:1919`）
  - 同一 session +0.4，同一主播 +0.2，编码一致 +0.1，间隔正常 +0.3
  - 间隔 >60min -0.3，编码不一致 -0.2
- 阈值决策：≥0.7 自动合并，0.4-0.7 通知，<0.4 跳过
- 编解码一致性检测 + 自动重编码
- 合并进度实时推送（WebSocket）
- 合并历史记录 + 节省空间统计

**后处理 Pipeline**
- 时间戳修复 → H.265 转码（可选）→ 智能重命名
- 高光检测（54 关键词 + 音频峰值 + 弹幕密度）
- 短视频生成（字幕/水印/弹幕叠加）
- 封面生成（多尺寸，基于高光帧）

**系统功能**
- 用户认证（email/password + session token）
- Tier 系统（Free/Pro/Team）+ 配额管理
- Discord/Telegram webhook
- 配置导入导出（JSON）
- 定时录制（按星期 + 时间段）
- 主播缩略图实时更新

### 2.2 半成品 / 未磨合的功能

| 功能 | 现状 | 问题 |
|------|------|------|
| **分发系统** | `distribute.py` 是 mock stub | 抖音/快手接口未实现，UI 有分发按钮但无实际效果 |
| **抖音录制** | 实验性标注 | ttwid cookie 依赖，API 频繁变化，稳定性差 |
| **合并确认流程** | 低信心度时只发 webhook | 前端没有"待确认合并"UI，用户不知道有任务等待 |
| **Storage 文件预览** | 有文件列表 | 无法内联播放，需要下载后才能看 |
| **Clips 批量操作** | 有下载 zip、导出本地 | 缺少批量选择、批量删除 UI |
| **高光规则管理** | API 完整（GET/POST/DELETE） | 前端规则 CRUD UI 不完整，用户无法方便地管理规则 |
| **字幕生成** | `subtitle_gen.py` 存在 | 触发时机不清晰，Storage 页面无"生成字幕"入口 |
| **录制文件缩略图** | 主播头像缩略图有 | 单个录制文件无关键帧缩略图 |

### 2.3 缺失的高价值功能

1. **合并待确认队列 UI** — 低信心度合并积压无处可见
2. **磁盘空间主动告警** — 只有 critical 时停录，没有预警 banner
3. **录制质量实时图表** — 码率数据已采集（`bandwidth_kbps`），但无可视化
4. **智能存储清理建议** — 基于重要性评分自动建议清理哪些文件
5. **全局搜索** — 跨主播搜索录制文件
6. **录制任务优先级** — 磁盘不足时优先保留哪些主播的录制

## 三、智能合并策略设计

### 3.1 现状问题

当前合并流程存在一个关键断层：

```
录制结束 → 计算信心度 → 0.4-0.7 → 发 webhook → 用户不知道
```

用户收到 webhook 通知后，需要手动进入 Storage 页面找到对应文件，操作路径长，体验差。

### 3.2 新策略：最小化用户操作

**决策树（保持现有阈值，补齐 UI 层）：**

```
录制结束
    ↓
计算合并信心度
    ├── confidence ≥ 0.7  →  自动合并（现有逻辑，保持）
    ├── 0.4 ≤ conf < 0.7  →  写入待确认队列 + 顶部 badge + Toast
    └── confidence < 0.4  →  跳过（现有逻辑，保持）
```

**待确认队列 UI（新增）：**
- 顶部导航栏右侧显示红色 badge（数字 = 待确认数量）
- 点击 badge 展开右侧抽屉（drawer）
- 抽屉内每条显示：
  - 主播名 + 平台图标
  - 片段数量 + 总时长
  - 最大时间间隔
  - 信心度进度条（颜色：橙色 = 中等信心）
  - 操作按钮：合并 / 忽略
- 抽屉顶部：全部合并 / 全部忽略

**智能分组逻辑增强：**
- 同一主播同一天，间隔 < 30min → 信心度 +0.1（额外加分）
- 跨天片段 → 信心度 -0.2（建议分开）
- 单片段时长 < 2min → 自动标记为垃圾片段，建议丢弃
- 编码不一致 → 自动重编码后合并（现有逻辑，保持）

### 3.3 后端变更

**新增 API：**
- `GET /api/merge-queue` — 获取待确认队列
- `POST /api/merge-queue/{session_id}/confirm` — 确认合并
- `POST /api/merge-queue/{session_id}/dismiss` — 忽略
- `POST /api/merge-queue/confirm-all` — 全部合并

**`recorder.py` 变更：**
- `auto_merge_for_model` 中，信心度 0.4-0.7 时，除发 webhook 外，同时写入 `merge_queue` 表
- 新增 `merge_queue` 数据库表（session_id, username, segments, confidence, created_at, status）

### P2 — 差异化功能（Sprint 3+）

#### P2-1: 全局搜索

跨主播搜索录制文件名、日期范围、时长范围。
- 后端：`server.py` 新增 `/api/search` 端点，SQLite FTS 或 LIKE 查询
- 前端：顶部搜索框，结果按主播分组展示
- **估时：** 1天

#### P2-2: 录制质量实时图表

码率折线图，帮助用户判断录制质量。
- 数据已有：`bandwidth_kbps` 通过 WebSocket 实时推送
- 前端：Streams 页面每个主播卡片添加迷你码率图（sparkline）
- **估时：** 1天

#### P2-3: 智能存储清理建议

基于"重要性评分"（高光密度 + 时长 + 最后访问时间）自动建议清理。
- 后端：新增评分算法，`/api/storage/cleanup-suggestions`
- 前端：Storage 页面"清理建议"面板，显示可安全删除的文件列表
- **估时：** 2天

#### P2-4: 分发系统真实实现

将 distribute.py 的 mock stub 替换为真实 API 调用。
- 抖音开放平台 API（需申请权限）
- 快手开放平台 API（需申请权限）
- **估时：** 3天+（依赖平台授权）

#### P2-5: 录制任务优先级

磁盘不足时，按优先级决定停止哪个主播的录制。
- 后端：`models` 表新增 `priority` 字段
- 录制器：磁盘 critical 时，停止最低优先级的录制
- 前端：主播设置面板添加优先级滑块
- **估时：** 1天
