# 字幕/翻译功能：集成 vs 独立产品 — 战略决策文档

> 创建时间：2026-04-29  
> 状态：待决策

---

## 一、核心问题

**字幕生成 + 字幕翻译，应该继续在 StreamVideo 内深化，还是切割成独立产品？**

---

## 二、市场格局

### 2.1 竞品现状

| 产品 | 定位 | 字幕策略 |
|------|------|---------|
| 剪映 | 国内短视频编辑 | 字幕免费，作为留存手段，不变现 |
| SubMagic | 国际短视频字幕 SaaS | 字幕是核心产品，€19-69/月 |
| Descript | 播客/视频编辑 | 字幕是附属功能，transcript-based editing 是差异化 |
| OpusClip | 长视频剪辑 | 字幕是剪辑工作流的一环，非独立 |
| 录播姬 | 国内直播录制 | **完全没有字幕功能** |
| olived.app | 国际直播录制 | 无字幕功能 |

**关键发现：直播录制后字幕处理，是一个完全未被覆盖的细分场景。**

### 2.2 市场规模

- 字幕翻译服务市场：2025 年 25 亿美元，CAGR 15%
- 字幕生成器市场：预计 2035 年达 45 亿美元，CAGR 9.9%
- 广告主字幕采用率：2 年内翻倍（2026 年数据）

### 2.3 技术壁垒

- **ASR（语音识别）**：壁垒低。Whisper 已开源，faster-whisper 性能接近商业产品，任何人可部署。
- **翻译质量**：壁垒中等。基础翻译已足够好，差异化在工作流集成而非翻译质量本身。
- **数据飞轮**：弱。不依赖私有数据，难以建立护城河。

---

## 三、StreamVideo 当前定位分析

### 3.1 核心价值链

```
直播监控 → 自动录制 → 智能合并 → 高光检测 → 短视频生成 → 分发
```

StreamVideo 的差异化在于这条**完整的垂直链路**，竞品（录播姬、olived）只做前两段。

### 3.2 字幕在链路中的位置

```
直播监控 → 自动录制 → 智能合并 → 高光检测 → 短视频生成（含字幕烧录）→ 分发
                                                        ↑
                                               字幕生成/翻译在这里
```

字幕功能目前服务于两个场景：
1. **短视频生成时的字幕烧录**（`clipgen.py`）— 深度集成，是短视频质量的核心
2. **录制文件的独立字幕生成**（`subtitle_gen.py`）— 松耦合，可独立触发

### 3.3 现有字幕代码资产

| 文件 | 功能 | 耦合度 |
|------|------|--------|
| `subtitle_gen.py` | Whisper ASR，SRT/VTT/ASS 生成 | 低（可独立调用） |
| `clipgen.py` | 字幕烧录到短视频 | 高（深度集成） |
| `recorder.py` | 自动字幕触发 | 中（通过配置开关） |

## 四、决策框架

### 4.1 独立成产品的条件

独立成产品需要满足：
1. 功能足够独立，不依赖宿主产品的核心数据
2. 目标用户群与宿主产品有明显差异
3. 独立后有更大的市场空间
4. 独立后的变现路径更清晰

### 4.2 逐条评估

**条件 1：功能独立性**

- `subtitle_gen.py` 只需要一个视频文件路径，完全独立
- 翻译功能只需要一个 SRT 文件，完全独立
- **结论：技术上可以独立，但没有必要**

**条件 2：用户群差异**

- StreamVideo 用户：直播录制者，需要完整的录制→剪辑→发布工作流
- 纯字幕工具用户：任意视频的字幕需求，不限于直播录制
- **结论：用户群有重叠，但字幕工具的潜在用户更广**

**条件 3：市场空间**

- 独立字幕 SaaS（SubMagic 模式）：国际市场有空间，国内因剪映免费效应极难变现
- 集成在 StreamVideo 内：服务现有用户，增强产品粘性，无需单独获客
- **结论：国内市场独立变现极难，集成更务实**

**条件 4：变现路径**

- 独立产品：需要重新建立品牌、获客、定价体系
- 集成功能：直接绑定 Pro/Team 套餐，提升 ARPU，无额外获客成本
- **结论：集成变现路径更短、更确定**

## 五、结论与建议

### 5.1 核心结论

**不应该独立成产品。字幕/翻译功能应该深化集成在 StreamVideo 内，作为"录制→发布"链路的关键差异化环节。**

理由：
1. **护城河在链路，不在单点**：StreamVideo 的价值是完整的工作流，字幕是其中一环。单独的字幕工具技术壁垒低，容易被复制。
2. **国内市场剪映效应**：独立字幕 SaaS 在国内几乎无法变现，集成在录制工具内反而是差异化。
3. **获客成本**：独立产品需要重新获客，集成功能直接服务现有用户。
4. **数据协同**：字幕与弹幕、高光检测、短视频生成有天然的数据协同（弹幕密度 → 高光 → 字幕烧录），独立后失去这个优势。

### 5.2 但可以做"功能模块化"

虽然不独立成产品，但字幕/翻译模块可以做成**可复用的独立模块**：

```
subtitle_gen.py     → 纯 ASR 引擎，无业务依赖
subtitle_translator.py → 纯翻译引擎，无业务依赖
```

这样未来如果需要，可以低成本地：
- 暴露为独立 API 服务
- 打包成 CLI 工具
- 集成到其他产品

## 六、功能深化路径

### 6.1 预览功能（独立于字幕）

预览功能与字幕是两个独立需求，应分开实现：

**预览（P0，已在 V4.0 路线图中）：**
- 改造 `/api/video` 端点支持 HTTP Range，实现 seek
- Storage 页面内联播放器
- 这是基础体验补齐，与字幕无关

**字幕（P1，在预览之后）：**
- 字幕生成触发入口（Storage 页面）
- 字幕翻译（新增 LLM 调用）
- 播放器内字幕轨道切换

### 6.2 字幕功能深化优先级

| 优先级 | 功能 | 价值 | 工作量 |
|--------|------|------|--------|
| P1 | Storage 页面字幕生成入口 | 补齐基础体验 | 0.5天 |
| P1 | HTTP Range 支持（预览 seek） | 基础体验 | 0.5天 |
| P2 | 字幕翻译（Claude Haiku） | 出海用户差异化 | 2天 |
| P2 | 播放器字幕轨道切换 | 完整体验 | 1天 |
| P3 | faster-whisper 替换 | 性能提升 | 1天 |
| P3 | 录制中实时预览（HLS） | 高级功能 | 1-2周 |

### 6.3 字幕翻译的成本控制

字幕翻译调用 LLM API，需要严格的成本控制：

| 策略 | 实现 |
|------|------|
| 默认 Claude Haiku | 成本 ~$0.05/小时视频，质量足够 |
| 批量翻译 | 每批 50 段，减少 API 调用次数 |
| 翻译缓存 | SQLite 缓存，相同文本只翻译一次 |
| 配额限制 | free: 0次/天，pro: 10次/天，team: 无限 |
| 用户自带 API Key | 允许用户配置自己的 Anthropic/OpenAI Key，成本转移 |

**最后一条"用户自带 API Key"是关键**：这样平台不承担 LLM 成本，用户自主控制质量和费用。

---

## 七、功能边界原则

**字幕功能只服务于 StreamVideo 自身录制的文件，不做通用字幕工具。**

- 不支持导入任意本地视频生成字幕
- 不支持上传外部文件
- 用户如需对外部视频生成字幕，推荐使用剪映/必剪

理由：支持本地文件导入会模糊产品定位，工程成本不低（上传 API、存储隔离、进度管理），而目标用户的视频本来就在 StreamVideo 的 recordings 目录内，不存在"导入"需求。

---

## 八、一句话总结

> 字幕/翻译不应该独立成产品，而应该成为 StreamVideo "录制→发布" 链路中的关键差异化环节。预览功能先行（P0），字幕翻译随后（P2），用户自带 API Key 解决成本问题。
